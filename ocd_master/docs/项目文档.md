# 2) 数据传输 / 加载 / 缓存建议

## 2.1 分层原则
- **Spectrum Analysis**：只前端临时态（session/local），不落库
- **Pre‑Recipe 起**：写入 `recipeSchema`（draft）
- **Recipe Build**：持续更新 `recipeSchema`
- **Run 之前**：后端校验 `recipeSchema` → 进入 queued/running

## 2.2 推荐缓存策略
- **LocalStorage**：保存当前 `draft schema` + 上次 selection（tool/wafer/time）
- **Memory**：页面内交互（dropdown/checkbox/plot）状态
- **后端**：存 `recipeSchema` 正式草稿（可版本化）

## 2.3 同步节奏
- **关键保存按钮**：立即 PUT schema
- **高频输入**：防抖（例如 500ms）或暂存至本地，保存时写入

# 3) API 设计与调用位置（前端）

下面是建议接口 + **调用位置**：

### 3.1 模型 & Workspace
- `GET /models/:modelId`
  - **调用位置**：Pre‑Recipe / Model 页面点击 Fetch
- `POST /runs`
  - **调用位置**：Recipe Setup 保存（生成 modelId + draft）
- `GET /runs/:runId/schema`
  - **调用位置**：进入任意 Pre‑Recipe / Recipe Build 页面初始化
- `PUT /runs/:runId/schema`
  - **调用位置**：Recipe Setup / CD Strategy / Fitting Strategy / Starting Point 保存

### 3.2 Spectrum / Pre‑Recipe
- `GET /spectra?tool&waferIds&timeRange`
  - **调用位置**：Spectrum Import
- `GET /sensitivity?splitTags&waferIds`
  - **调用位置**：Sensitivity Start
- `GET /precision?waferId&pointId`
  - **调用位置**：Precision Start
- `POST /pre-recipe/commit`
  - **调用位置**：Recipe Setup 保存（落库 dataset + baseline）

### 3.3 Template
- `GET /templates`
  - **调用位置**：Platform / Templates 列表
- `GET /templates/:templateId`
  - **调用位置**：Template Preview
- `POST /templates`
  - **调用位置**：Results Summary “Save to Template”

### 3.4 Template Enable 校验
- `POST /models/:modelId/template/validate`
  - **调用位置**：Model 页面启用 Template 前
- `POST /runs/:runId/template/apply`
  - **调用位置**：校验通过后，覆盖 recipeBuild

# 一、Postgres 方案（推荐：事务 & 可审计）

## 1) 核心表结构

### `workspaces`
- `id` (pk) → `modelId` 或 workspaceId
- `type` (temp/model)
- `status` (temp/draft/queued/running/completed)
- `owner`
- `recipe_name`
- `project`
- `product_id`
- `version`
- `created_at`
- `updated_at`

**索引**
- `(status)`, `(owner)`, `(project)`, `(product_id)`, `(updated_at)`

---

### `models`
- `model_id` (pk)
- `basis_cd` (jsonb)
- `constraint_cd` (jsonb)
- `materials` (jsonb)
- `created_at`
- `updated_at`

**索引**
- `(model_id)` + 可选 `GIN` on `materials` for JSONB search

---

### `recipe_schemas`
- `id` (pk)
- `workspace_id` (fk → workspaces.id)
- `schema` (jsonb) → 全量 schema
- `version` (text)
- `is_active` (bool)
- `created_at`
- `updated_at`

**索引**
- `(workspace_id, is_active)`, `(workspace_id, version)`

---

### `templates`
- `template_id` (pk)
- `template_name`
- `model_id`
- `recipe_meta` (jsonb)
- `schema` (jsonb)
- `created_at`
- `updated_at`
- `owner`

**索引**
- `(model_id)`, `(owner)`, `(updated_at)`

---

### `checkpoints`
- `checkpoint_id` (pk)
- `model_id`
- `version`
- `name`
- `schema_snapshot` (jsonb) → 可选
- `kpi_snapshot` (jsonb)
- `created_at`

**索引**
- `(model_id, version)`, `(model_id, name)` unique for name+version

---

## 2) 典型查询
- **Template 列表**：`SELECT template_id, template_name, model_id, recipe_meta FROM templates`
- **Recipe schema load**：`SELECT schema FROM recipe_schemas WHERE workspace_id = ? AND is_active = true`
- **Checkpoint 列表**：`SELECT * FROM checkpoints WHERE model_id=? AND version=?`

# 三、选型建议

- **Postgres** 更适合：  
  - 强事务一致性（schema 更新可审计）  
  - 复杂筛选（Assets/Template/History）
- **Mongo** 更适合：  
  - schema 变化频繁  
  - 需要快速存整块 JSON

如果你希望“模板 / schema / checkpoints”全是 JSON 直存，可用 Mongo；  
如果你希望“资产检索 / 权限 / 版本管理”强一致，建议 Postgres。
