## 当前实现逻辑（2026-03）

> 本节描述**当前代码已实现**的优化链路（以 `ocd_algorithm_api/ocd_auto_opt/pipeline/optimizer.py` 为准）。

### 1. Coupling 入口规则

- 配置来源：`recipe_schema.cdStrategy.schemes[].expression`
- 解析函数：`utils/model_utils.py::coupling_expressions`
- 候选生成：`core/coupling.py::coupling_candidates`

当前行为：
- 永远会先加入一个空表达式分支 `""`（baseline 分支）
- 再追加配置中的有效 coupling 表达式

这意味着：
- **如果没有任何 coupling 表达式**，最终只会得到一个分支 `[""]`
- 该分支执行 `apply_coupling(base_model, "")`，会直接返回原始 `base_model` 的深拷贝
- 即：**无 coupling 时按原始 model_json 正常跑完整优化流程**

---

### 2. 优化主流程（按 `optimizer.run()`）

1. 解析输入与路径  
- 解析 baseline/regression/precision 光谱路径  
- 解析 seed/fitting/precision/sensitivity/final_regression 的配置  

2. Coupling 循环  
- 对每个 coupling 分支生成 `coupled_model`

3. Seed Search  
- 使用 baseline 光谱进行候选材料组合评估  
- 调 `get_result` 做拟合指标评估（GOF/Residual/Correlation/LBH）  
- 调 `getSpectrum` 生成 simulated 光谱并计算 plain MSE  
- 产出 topK seed，落盘到 `Results/seed_search/...`

4. Fitting（每个 seed）  
- 按 `iteration -> material -> step` 在 baseline 光谱上拟合  
- GOF 提升才接受，否则回退  
- 落盘到 `Results/fitting/...`

5. Precision Check  
- 使用 precision 光谱集合评估 LBH 与 target CD 的 3σ  
- 包含 baseline/1D/2D case 选择  
- 落盘到 `Results/precision/...summary.json`

6. Sensitivity  
- 计算 baseline 与 baseline± 的差异，输出总 sensitivity 与分段权重  
- 落盘到 `Results/sensitivity/...`

7. Final Regression（每个 grid）  
- 每个 grid 组合分别进入最终回归阶段  
- 在迭代中同时检查 baseline GOF 与 KPI 判定  
- 命中有效解后加入 valid solution 集合，可按 top_n 提前停止  
- 落盘到 `Results/final_regression/...`

8. 汇总  
- 输出 `Results/optimization_result.json`

---

### 3. 前端 Trace 对齐规则

Trace 页按阶段分 tab：
- `seed_search`

# 2) 数据传输 / 加载 / 缓存建议

## 2.1 分层原则
- **Spectrum Analysis**：只前端临时态（session/local），不落库
- **Pre‑Recipe 起**：写入 `recipeSchema`（draft）
- **Recipe Build**：持续更新 `recipeSchema`
- **Run 之前**：后端校验 `recipeSchema` → 进入 queued/running

## 2.2 推荐缓存策略
- **LocalStorage**：保存当前 `draft schema` + 上次 selection（tool/wafer/time）
- **Memory**：页面内交互（dropdown/checkbox/plot）状态
- **后端**：存 `recipeSchema` 正式草稿（可版本化）

## 2.3 同步节奏
- **关键保存按钮**：立即 PUT schema
- **高频输入**：防抖（例如 500ms）或暂存至本地，保存时写入

# 一、Postgres 方案（推荐：事务 & 可审计）

## 1) 核心表结构

### `workspaces`
- `id` (pk) → `modelId` 或 workspaceId
- `type` (temp/model)
- `status` (temp/draft/queued/running/completed)
- `owner`
- `recipe_name`
- `project`
- `product_id`
- `version`
- `created_at`
- `updated_at`

**索引**
- `(status)`, `(owner)`, `(project)`, `(product_id)`, `(updated_at)`

---

### `models`
- `model_id` (pk)
- `basis_cd` (jsonb)
- `constraint_cd` (jsonb)
- `materials` (jsonb)
- `created_at`
- `updated_at`

**索引**
- `(model_id)` + 可选 `GIN` on `materials` for JSONB search

---

### `recipe_schemas`
- `id` (pk)
- `workspace_id` (fk → workspaces.id)
- `schema` (jsonb) → 全量 schema
- `version` (text)
- `is_active` (bool)
- `created_at`
- `updated_at`

**索引**
- `(workspace_id, is_active)`, `(workspace_id, version)`

---

### `templates`
- `template_id` (pk)
- `template_name`
- `model_id`
- `recipe_meta` (jsonb)
- `schema` (jsonb)
- `created_at`
- `updated_at`
- `owner`

**索引**
- `(model_id)`, `(owner)`, `(updated_at)`

---

### `checkpoints`
- `checkpoint_id` (pk)
- `model_id`
- `version`
- `name`
- `schema_snapshot` (jsonb) → 可选
- `kpi_snapshot` (jsonb)
- `created_at`

**索引**
- `(model_id, version)`, `(model_id, name)` unique for name+version

---

## 2) 典型查询
- **Template 列表**：`SELECT template_id, template_name, model_id, recipe_meta FROM templates`
- **Recipe schema load**：`SELECT schema FROM recipe_schemas WHERE workspace_id = ? AND is_active = true`
- **Checkpoint 列表**：`SELECT * FROM checkpoints WHERE model_id=? AND version=?`

# 三、选型建议

- **Postgres** 更适合：  
  - 强事务一致性（schema 更新可审计）  
  - 复杂筛选（Assets/Template/History）
- **Mongo** 更适合：  
  - schema 变化频繁  
  - 需要快速存整块 JSON

如果你希望“模板 / schema / checkpoints”全是 JSON 直存，可用 Mongo；  
如果你希望“资产检索 / 权限 / 版本管理”强一致，建议 Postgres。
